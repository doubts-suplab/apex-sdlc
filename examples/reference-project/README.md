# APEX Reference Journey — Customer Refunds Service

A **worked, runnable example** of one project traversing all seven APEX SDLC phases, told from each
persona's point of view. Every artifact under [`artifacts/`](artifacts/) was **generated by the governed
agent-harness**, not hand-written — regenerate them any time, offline, with no API key:

```bash
cd platform/backend
python -m venv .venv && . .venv/bin/activate && pip install -e .
LLM_PROVIDER=stub python -m app.demo.reference_journey
```

The project: a **Customer Refunds Service** (Spring Boot microservice) built from a one-paragraph
[brief](brief.md). It is deliberately small and relatable, and it exercises the parts of the framework
that matter — PII handling, quality gates, and governance.

---

## The journey at a glance

Each phase is run by an AI agent on the [agent-harness](https://github.com/doubts-suplab/agent-harness).
The agent **proposes** a decision; the harness **governs** whether it may auto-enforce. The single most
important thing to notice: the four **SUGGEST**-authority phases can *never* auto-enforce (harness gate
rule G-5), so they always route to a human — this is APEX's *"AI drafts, humans approve"* principle
expressed directly by the authority ladder, not re-coded in the app.

| # | Phase | Persona | Agent | Authority | Decision | Conf. | Governed outcome |
|---|---|---|---|---|---|---|---|
| 1 | Requirements | Business Analyst | `RequirementsAgent` | SUGGEST | SUGGEST | 0.86 | → **human review** |
| 2 | Architecture | Architect | `ArchitectureAgent` | SUGGEST | SUGGEST | 0.84 | → **human review** |
| 3 | Development | Developer | `PRReviewerAgent` | ALERT | ALERT | 0.88 | **auto-enforced** advisory |
| 4 | Testing | QA Engineer | `QAAnalystAgent` | SUGGEST | SUGGEST | 0.85 | → **human review** |
| 5 | CI/CD | Tech Lead | `ReleaseEngineerAgent` | RATE_LIMIT | ALLOW | 0.88 | **auto-enforced** release |
| 6 | Docs | Developer | `TechWriterAgent` | SUGGEST | SUGGEST | 0.83 | → **human review** |
| 7 | Governance | CISO | `ComplianceOfficerAgent` | BLOCK | ALERT | 0.60 | → **human review** |

**Totals:** 7 phases · 17 artifacts · 2 auto-enforced · 5 routed to a human · **0 confidence-gate bypasses**
· 7 append-only audit entries. (Live values in [`journey.json`](journey.json).)

## Phase by phase — who does what

### 1 · Requirements — *Business Analyst*
The **BA** hands in the brief. `RequirementsAgent` drafts Gherkin stories and a gap analysis. Because it
is a SUGGEST agent, the harness routes the stories to the BA for approval — marked `[AI-Draft]` until a
human accepts. **Hands off to:** the Architect.
→ [`user-stories.md`](artifacts/requirements/user-stories.md) · [`gap-analysis.md`](artifacts/requirements/gap-analysis.md)

### 2 · Architecture — *Architect*
The **Architect** takes the approved stories. `ArchitectureAgent` proposes an ADR (standalone hexagonal
service, idempotent refunds, outbox to the ledger), a component diagram, and a tech-debt note. SUGGEST →
the ADR is *Proposed*, awaiting architect sign-off. **Hands off to:** the Developer.
→ [`ADR-0001-refund-service.md`](artifacts/architecture/ADR-0001-refund-service.md) · [`component-diagram.mmd`](artifacts/architecture/component-diagram.mmd) · [`tech-debt.md`](artifacts/architecture/tech-debt.md)

### 3 · Development — *Developer*
The **Developer** opens a PR. `PRReviewerAgent` (ALERT authority) reviews the diff and finds real issues
— a SQL-injection risk and PII being logged. At 0.88 confidence the harness **auto-enforces** the advisory
review (it gets posted), but a human still merges. **Feeds:** the Governance risk register. **Hands off to:** QA.
→ [`pr-review.md`](artifacts/development/pr-review.md) · [`quality-report.md`](artifacts/development/quality-report.md)

### 4 · Testing — *QA Engineer*
The **QA Engineer** takes the stories and the code. `QAAnalystAgent` drafts a test plan, a BDD case suite
mapped to the stories, and a coverage-gap analysis (it flags the same `already_refunded` branch the PR
review missed a test for). SUGGEST → routed to QA for sign-off. **Hands off to:** the Tech Lead.
→ [`test-plan.md`](artifacts/testing/test-plan.md) · [`test-cases.md`](artifacts/testing/test-cases.md) · [`coverage-gaps.md`](artifacts/testing/coverage-gaps.md)

### 5 · CI/CD — *Tech Lead*
The **Tech Lead** cuts the release. `ReleaseEngineerAgent` (RATE_LIMIT) assembles release notes, a
deployment checklist, and a rollback plan. With checks green at 0.88 (≥ 0.85 threshold) the harness
**auto-enforces** the release decision. Had checks been red, it would have raised an ALERT to hold.
→ [`release-notes.md`](artifacts/cicd/release-notes.md) · [`deployment-checklist.md`](artifacts/cicd/deployment-checklist.md) · [`rollback-plan.md`](artifacts/cicd/rollback-plan.md)

### 6 · Docs — *Developer / Technical Writer*
The **Developer** wearing a tech-writer hat ships the docs. `TechWriterAgent` drafts an API reference and
an onboarding guide. SUGGEST → a human publishes them. **Consumed by:** future developers and the BA.
→ [`api-reference.md`](artifacts/docs/api-reference.md) · [`onboarding-guide.md`](artifacts/docs/onboarding-guide.md)

### 7 · Governance — *CISO*
The **CISO** gets oversight of the whole change. `ComplianceOfficerAgent` (BLOCK authority) weighs the two
golden-rule breaches the PR review flagged at a moderate composite risk (0.60). BLOCK only auto-enforces at
≥ 0.95, so this **material-but-uncertain** case is routed to a human — exactly the right call. It emits a
governance/audit report and a risk register.
→ [`governance-report.md`](artifacts/governance/governance-report.md) · [`risk-register.md`](artifacts/governance/risk-register.md)

## What this example proves

1. **Every persona is served** — each of the seven personas owns or consumes a phase, with a concrete
   agent and concrete artifacts (see [`docs/personas.md`](../../docs/personas.md)).
2. **The framework is demonstrable, not just described** — one command regenerates all 17 artifacts
   through the governed harness.
3. **Governance is real and centralized** — enforcement is decided by the harness confidence gate, not by
   the agents; low-confidence and SUGGEST decisions route to humans; the gate is never bypassed.
4. **APEX leverages, it does not rebuild** — the agents are thin (`platform/backend/app/agents/`); the
   gate, tool registry, audit, human-review routing, and safe-failure defaults all come from
   [agent-harness](https://github.com/doubts-suplab/agent-harness).

See the same walk rendered visually in [`docs/reference-journey.html`](../../docs/reference-journey.html),
and the live API at `GET /api/v1/journey/reference` (run the backend with `LLM_PROVIDER=stub`).
