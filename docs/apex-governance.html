Section 1

Purpose & Scope

This document governs the use of AI coding assistants (Claude Code, GitHub Copilot) and AI automation pipelines within projects operating under the APEX framework. It protects the organisation from data leakage, model hallucinations entering production, intellectual property risk, and non-compliant AI-generated content.

Scope Applies to all team members using AI tools on APEX-governed projects — developers, BAs, QA engineers, programme managers, agile coaches, and any automation pipelines running Claude or Copilot.

Core Principles

  • AI generates drafts; humans approve and commit. No AI output reaches production without human sign-off.
  • Every AI action is logged. Full audit trail in your own AWS infrastructure — nothing leaves your environment.
  • Least privilege by default. Capabilities expand only after each gate is explicitly passed and ARB-approved.
  • Graceful degradation. If any AI service is unavailable, the SDLC pipeline continues — AI steps skip with a warning, never block delivery.

How these principles are enforced — the agent-harness

APEX does not re-implement agent governance. Every phase agent runs on the generic agent-harness, which owns the controls below. An agent proposes a decision and a confidence; the harness decides whether it may auto-enforce. This makes "AI generates drafts; humans approve" a property of the runtime, not a convention.

The confidence gate & authority ladder Centralized in the harness core, running on every invocation, and impossible to disable. It is the only place a decision's auto_enforced flag is set — an agent can never set it. Authority levels: OBSERVE < SUGGEST < ALERT < RATE_LIMIT < BLOCK. SUGGEST-authority phases can never auto-enforce (they always route to a human); ALERT/RATE_LIMIT/BLOCK auto-enforce only at or above their confidence threshold (0.80 / 0.85 / 0.95). Any failure resolves to a safe, non-enforcing decision. The confidence_gate_bypass_total counter must always be 0.

Section 3

Data Classification

3.1 — MUST NOT be pasted into any AI tool

Prohibited data Production database dumps or exports · Customer names, email addresses, national IDs, financial data (PII/PCI) · API keys, secrets, passwords, or AWS credentials · Proprietary algorithms marked CONFIDENTIAL or RESTRICTED · Unpublished financial results or M&A information · Third-party source code under restrictive licences

3.2 — MAY be used with standard tools (Claude.ai / Copilot)

  • Anonymised or synthetic data (real field names, fake values)
  • Internal business logic that is not PII-adjacent
  • Architecture diagrams and design documents (non-sensitive)
  • Test data with no connection to real individuals

3.3 — Requires the enterprise-grade Claude API (not consumer Claude.ai)

  • Any code that touches customer records
  • Automated pipelines processing sprint/Jira data
  • Anything logging outputs to shared systems (Confluence, Slack)
Why Enterprise API data is not used for model training. Consumer Claude.ai inputs may be used for improvement unless opted out at the organisation level. Always use the API for any data that isn't purely synthetic.
Section 4

Code Review Gates

All AI-generated code must pass the following five gates before merging. AI-generated commits are automatically labelled ai-assisted by the PR workflow.

1
Automated PR Review
GitHub Actions ai-pr-review.yml runs on every PR — OWASP scan, diff analysis, auto-generated description.
2
Human Reviewer
At least one human must review AI-generated files, not just approve the automated check. Mandatory 2 reviewers for ai-assisted PRs.
3
Security Scan — No HIGH Findings
ai-pr-review.yml OWASP check must return zero HIGH severity findings before merge is permitted.
4
Golden Rules Check
eeik_bootstrap rules enforced: no @Autowired on fields, no javax.* in Boot 3.x, no SELECT *, constructor injection only.
5
Test Coverage Gate
JaCoCo (Java) / Istanbul (Angular) must report ≥ 80% line coverage on all files changed in the PR.
Section 5

PII Guard

The PII Guard is a two-layer pre-prompt scanner that runs on every API call in the automation pipeline. It prevents sensitive data from ever reaching an external model.

Layer 1 — Regex Patterns

Fast, deterministic scanning for 14 pattern categories applied locally before the API call:

PatternExamples matchedAction on hit
EMAILuser@domain.comBlock + alert
NINO / SSNAB123456C · 123-45-6789Block + alert
CREDIT_CARDLuhn-valid 16-digit numbersBlock + alert
BANK_ACCOUNT / SORT_CODEUK sort code + accountBlock + alert
AWS_ARN / AWS_ACCOUNT_IDarn:aws:…Block + alert
SECRET_HEURISTIC32+ char high-entropy stringsBlock + alert
UK_POSTCODE / US_ZIPSW1A 1AA · 90210Log only
DATE_OF_BIRTHDOB patterns in contextLog only

Layer 2 — AWS Comprehend

For anything that passes regex, AWS Comprehend DetectPiiEntities provides ML-based detection with a confidence threshold of 0.90. Text is chunked below the 5000-byte API limit. Findings from both layers are deduplicated by span overlap before the decision is made.

Current status PII Guard regex layer is active (Phase 0). AWS Comprehend integration is planned for Phase 2 (Week 15). Until then, high-confidence regex patterns block requests; lower-confidence patterns log-only. See POL-005.

Scrubbing output

When a finding is logged-only (not blocked), matched spans are replaced with [REDACTED:TYPE] in the stored audit log. The original prompt is never stored.

Section 6

Mainframe Gate Policy

Mainframe systems (IBM COBOL, CICS, DB2 z/OS, IBM i) carry the highest risk in any AI-augmented SDLC. A single incorrect generated instruction can trigger batch abends, corrupt VSAM datasets, or silently produce wrong financial outputs. This gate controls how AI assistance expands on mainframe work.

Phase 0 — Analysis Only

ACTIVE

Default and current state. ARB approval required to advance to Phase 1.

✅ AI MAY
  • Explain existing COBOL programs in plain English
  • Decode JCL steps and VSAM structures
  • Convert COBOL copybooks to Java DTOs (documentation only)
  • Generate runbooks from existing programs
  • Assess migration complexity scoring
  • Produce data flow diagrams from batch job chains
🚫 AI MUST NOT
  • Generate new COBOL, JCL, or RPG code
  • Suggest modifications to existing programs
  • Produce DB2 DDL or DML for z/OS
  • Generate CICS transaction definitions
  • Create or modify VSAM cluster definitions
  • Recommend changes to sort cards or DFSORT parameters

Phase 1 — Assisted Analysis + Draft Generation

Pending — Week 17 gate

All five Week 17 criteria must be met before ARB will consider approval:

  • ≥ 20 COBOL programs documented via Phase 0 AI analysis
  • Mainframe dev team has completed APEX AI literacy training (2-hour session)
  • At least 2 copybook-to-Java-DTO conversions validated end-to-end by mainframe architect
  • PII Guard enabled and tested on all mainframe-adjacent prompts
  • Security guild has reviewed mainframe prompt library and approved

Once unlocked, AI may additionally generate draft COBOL with explicit human-review requirement, suggest JCL modifications (diff-style only), generate RPG IV stubs, and produce read-only DB2 SELECT drafts.

Phase 2 — Full Assisted Modernisation

Future — post Phase 1 + ARB

Triggered after successful Phase 1 for a minimum of 2 sprints with ARB sign-off. Unlocks Spring Batch job generation from COBOL analysis, Liquibase migration scripts from DB2 DDL, Strangler Fig MQ bridge stubs, and full migration readiness reports. All require mainframe architect sign-off and parallel-run testing for ≥ 2 weeks.

Section 7

Audit & Logging

The APEX governance pipeline logs every AI-assisted action. All logs stay in your own AWS infrastructure — nothing is sent to external systems.

EventLog destinationRetention
PR review AI outputGitHub Actions log + S3 audit bucket90 days
Claude API call (Jira bridge)CloudWatch Logs /apex/automation30 days
PII scan result (all outcomes)S3 audit bucket (encrypted, SSE-S3)1 year
ARB approval / rejectionConfluence ARB log pageIndefinite
Mainframe Gate statusgovernance/mainframe-gate-log.mdIndefinite
Prompt hash (not plaintext)DynamoDB apex-audit table90 days
What is logged For each API call: timestamp, user identity, SHA-256 prompt hash (never plaintext), model used, token count, output accepted/modified/rejected, and any PII scan findings. Plaintext prompts are never stored.
Section 8

Hallucination Risk Controls

AI models can produce plausible-sounding but incorrect outputs. The following controls are mandatory for all AI-assisted work:

RiskControl
Invented API contractsAlways validate generated OpenAPI specs against the running service
Wrong business rules in generated testsBA must sign off on acceptance criteria before tests are considered passing
Security vulnerabilities in generated codeSecurity guild spot-check of PRs labelled security-review-needed
Outdated dependency versionsDependabot + mvn versions:display-dependency-updates on every merge
Invented architecture decisionsAll AI-suggested ADRs go through ARB before adoption — never commit directly
Prompt discipline Always end every prompt with: "Review this output for accuracy and flag anything you're uncertain about." This is Prompt #10 in the APEX library for a reason — it surfaces model uncertainty before humans review.
Section 9

Intellectual Property

  • Ownership: AI-generated code contributed to a company project is owned by the company, subject to the same IP assignment as human-authored code.
  • Copyright contamination: If Copilot suggests a block of more than 150 characters that appears verbatim in open-source code, reject it and rewrite manually. Use Copilot's "Show references" feature to check provenance before accepting.
  • Model provider terms: Anthropic and GitHub Copilot Business terms permit enterprise customers to own their outputs. Verify before using any third-party AI tool not listed in this policy.
  • Attribution: AI-generated files carry the ai-assisted PR label and are identifiable in audit logs. This does not affect copyright — the committing developer remains the author of record.

Section 11

AI Review Board (ARB)

The ARB meets monthly and is the governing body for all AI-related decisions on APEX projects. It is the only body authorised to approve gate transitions, new tool additions, and policy changes.

Responsibilities

  • Approving new AI tools for use on APEX projects
  • Reviewing HIGH-severity violations and determining remediation
  • Signing off on Mainframe Gate transitions (§6)
  • Reviewing and updating this policy quarterly
  • Reviewing usage metrics and token spend trends
  • Approving expansion of permitted use categories
